# 問3 解説

## 問題の概要
C社は産業用機械メーカで、20年以上前に構築した**基幹システム**の再構築プロジェクトを進めています。このプロジェクトは、要件定義から結合テストまでをP社に外部委託しています。現在、**結合テストの終了段階**にあり、監査部では最終のシステムテスト開始前に、結合テスト完了評価の適切性についてシステム監査を実施することになりました。監査人D氏が中心となり、関係者へのヒアリングや資料閲覧を通じて監査を進めます。

## 出題のポイント
本問題では、**システム開発を外部委託している場合の結合テスト完了評価**に対するシステム監査の観点と、具体的な監査手続を立案する能力が問われています。特に、結合テストの**品質評価（不良密度、テスト密度）**、**課題管理**、**システム間インターフェースの確認**、および**プロジェクト体制の妥当性**という多角的な視点から、テストの完了基準が適切に満たされているかを評価する知識とスキルが求められます。また、外部委託先との責任分担や、システムテストへの影響を考慮した監査判断が重要となります。

## 設問ごとの解説

### 設問1
〔追加の確認事項と検討事項〕(1)について, D氏が不十分と判断した理由は何か。30字以内で述べよ。

設問1は、D氏が「テスト項目数が十分であること」だけでは不十分と判断した理由を答えるものです。
〔追加の確認事項と検討事項〕(1)には、「販売管理システムの品質評価結果では、問題がない理由としてテスト項目数が十分であることを評価しているが、それだけでは不十分なので、追加で確認する」とあります。
表1「結合テスト完了基準と根拠資料」の項番2（品質）には、完了基準として「不良密度（不良発生件数/システム規模）」と「テスト密度（テスト項目数/システム規模）」が挙げられています。
販売管理システムの品質評価結果では、「テスト密度が基準範囲内であり、テスト項目数が十分であること」とされていますが、D氏がこれを不十分と判断した背景には、**テスト項目の数だけでは品質は保証できない**という考えがあります。重要なのは、テスト項目の「量」だけでなく、その「質」です。例えば、重要な機能や複雑なロジックに対するテストが十分に計画・実施されているか、エラーケースや境界値のテストが考慮されているかなど、テスト項目の**網羅性や深度**が問題となります。

> **正解**: テスト項目の質が十分であることが評価されていないから

**思考プロセス**:
1.  〔追加の確認事項と検討事項〕(1)でD氏が「テスト項目数が十分」という評価を不十分としている点に着目します。
2.  表1の品質に関する完了基準には「不良密度」と「テスト密度」があります。テスト密度は「テスト項目数/システム規模」で量を評価する指標です。
3.  テスト項目数が多くても、それが表面的なテストばかりであったり、重要な業務ロジックやリスクの高い部分に十分なテストが割り当てられていなかったりすれば、品質は保証されません。したがって、D氏は**テスト項目の内容や網羅性、すなわち「質」**が評価されていないことを問題視していると判断できます。
4.  この理由を30字以内でまとめます。

**誤答しやすいポイント・注意点**:
*   単に「テスト項目が不足している」と答えるのは誤りです。問題文は「テスト項目数が十分である」と記述しています。D氏が不十分としたのは「質」の観点です。
*   採点講評でも「テスト項目の質が十分であることが評価されていないから」が解答例とされており、「テスト項目の質」に着目できるかがポイントです。

### 設問2
〔追加の確認事項と検討事項〕(2)について、対応が適切に行われているかどうかを確認するための監査手続は何か。監査証拠を含めて50字以内で具体的に述べよ。

> **正解**: 課題管理表を閲覧し、マスタデータの作成ミスの再発防止策の内容と対応期限を確かめる。

**思考プロセス**:
1.  〔追加の確認事項と検討事項〕(2)と「PMへのヒアリング」(3)の「マスタデータの作成ミス」に着目します。これが、確認すべき「生産管理システムで発生した課題」です。

3.  特に「マスタデータの作成ミス」という具体的な原因が示されているため、その再発防止策の内容と、それがいつまでに完了する予定か（対応期限）を確認することが重要です。
4.  これらの監査手続と確認すべき内容を監査証拠（課題管理表）と合わせて50字以内でまとめます。

**誤答しやすいポイント・注意点**:
*   単に「是正処置計画書を確認する」といった抽象的な解答では不十分です。**マスタデータ作成ミス**という具体的な原因に焦点を当て、その再発防止策の内容まで踏み込んで記述することが求められます。

〔追加の確認事項と検討事項〕(3)について、プロジェクトとしての対応に関して何を確認すべきか。35字以内で述べよ。

設問3は、プロジェクトとしての対応に関して何を確認すべきかを答えるものです。
〔追加の確認事項と検討事項〕(3)には、「生産管理システムの品質評価結果に記載されている類似不良の品質向上策に関して、**財務会計システムは他業務システムともインタフェースがある**ことから、プロジェクトとしての対応が適切かどうかを確認する」とあります。
「生産管理システムの品質評価結果」の(2)には、「チーム内で類似不良についての品質向上策を実施済み」とありますが、財務会計システムが他の業務システムとも連携しているという情報が追加されています。
このことから、生産管理システム内で実施された品質向上策が、財務会計システムを介して連携する**他のシステム**にも影響を及ぼす可能性があると考えられます。したがって、類似不良に関する知見や品質向上策が、生産管理チーム内だけでなく、**関連する他の開発チーム間**でも適切に共有され、必要に応じて対策が講じられているかを確認することが、プロジェクトとしての適切な対応と言えます。

> **正解**: 類似不良点検について、開発チーム間で情報共有されているか

**思考プロセス**:
1.  〔追加の確認事項と検討事項〕(3)の「財務会計システムは他業務システムともインタフェースがある」という記述に着目します。
2.  「生産管理システムの品質評価結果」(2)には「チーム内で類似不良についての品質向上策を実施済み」とありますが、インターフェースがある以上、その対策が生産管理チーム内にとどまるだけでは不十分な可能性があります。
3.  プロジェクト全体としての対応を考えるならば、関連するシステム間（開発チーム間）での**情報共有**と、必要に応じた**横断的な対策**が重要です。
4.  したがって、類似不良に関する知見が開発チーム間で共有されているかを確認すべきであると判断できます。これを35字以内でまとめます。

**関連する重要な技術知識・用語の補足説明**:
*   **システム間連携（インターフェース）**: 複数のシステムがデータをやり取りする仕組み。連携するシステム間で問題が発生した場合、影響範囲が広がる可能性があり、全体的な対策と情報共有が不可欠です。
*   **部門横断的な課題解決**: 大規模なプロジェクトでは、特定のチーム内で解決した問題が、他のチームや連携システムに影響する場合があるため、部門やチームを越えた情報共有と連携が求められます。
*   採点講評でも「財務会計システムは生産管理システムだけでなく他業務システムともインタフェースがあるということを踏まえ,生産管理開発チーム内だけでなく開発チーム間での情報共有が必要であることに気付いてほしい。」と指摘されています。

### 設問4
〔追加の確認事項と検討事項〕(4)について、改善提案の内容として考えられることは何か。50字以内で具体的に述べよ。

設問4は、購買管理システムと生産管理システムとのインタフェース機能の残課題について、改善提案の内容として考えられることを答えるものです。
〔追加の確認事項と検討事項〕(4)には、「購買管理システムと生産管理システムとのインタフェース機能の残課題について、システムテストでのリスクを低減するための改善提案を検討する」とあります。
「購買管理システムの課題管理表」の(3)には、「生産管理システムとのインタフェース機能の仕様漏れについて、“プログラム修正とチーム内のテストは完了済み。**生産管理システムと接続させての確認はシステムテストで実施予定**。”と記述されていた」とあります。
ここで問題となるのは、**システムテストで初めて**二つのシステムを接続して確認する計画になっている点です。システムテストは、全てのシステムを統合して行う最終段階のテストであり、そこでインターフェースの問題が発覚すると、影響範囲が大きく、手戻りや遅延のリスクが高まります。システムテストでのリスクを低減するためには、それ以前の段階、つまり**システムテスト開始前**に、対象となる**二つのシステム間での疎通確認テスト**を実施し、インターフェースの問題を早期に発見・解決しておくことが効果的な改善提案となります。

> **正解**: システムテスト開始前に購買管理システムと生産管理システムとの疎通確認テストを実施すること

**思考プロセス**:
1.  〔追加の確認事項と検討事項〕(4)の「インタフェース機能の残課題」と「システムテストでのリスクを低減するための改善提案」に着目します。
2.  「購買管理システムの課題管理表」(3)の記述を読みます。「生産管理システムと接続させての確認はシステムテストで実施予定。」という点が、システムテストでのリスクを高める要因であると判断できます。
3.  システムテストは複数のシステムを連携させるテストであり、個別のシステム間の連携不備は、その前の段階で解決しておくべきです。
4.  したがって、システムテストに移行する前に、問題となっている**購買管理システムと生産管理システムとの間で、先に疎通確認テストを実施**するという提案が有効です。
5.  この改善提案を50字以内で具体的に記述します。

**関連する重要な技術知識・用語の補足説明**:
*   **疎通確認テスト**: システム間のインターフェースが正しく機能し、データが正常にやり取りできることを確認するテスト。結合テストやシステムテストの初期段階で実施されることが多く、問題を早期に発見する上で重要です。
*   **テスト工程の階層性**: 単体テスト → 結合テスト → システムテスト → 受け入れテストと、テストは段階的に実施されます。下位のテスト（単体・結合）で問題を解決しておくことで、上位のテスト（システム・受け入れ）での手戻りやリスクを低減できます。

### 設問5
〔追加の確認事項と検討事項〕(5)について、当プロジェクトの体制に関して何を確認すべきか。45字以内で述べよ。

設問5は、今後のシステムテストにおけるプロジェクト体制に関して何を確認すべきかを答えるものです。
〔追加の確認事項と検討事項〕(5)には、「〔工程計画〕の内容を踏まえ、今後のシステムテストにおける当プロジェクトの体制に問題がないかどうかを確認する」とあります。
「工程計画」には、「C社は、業務システムごとの結合テストを完了した後、**システムテストにおいて、業務システム全てを連携させたテストとユーザ受入テストを行う計画**である」とあります。
「開発体制」には、「システム部では、現行システムの運用と保守を中心に行ってきたので、新規のシステム開発のノウハウが不足している。そのため、実質的には、要件定義の段階から**P社が中心となって開発プロジェクトを進めており**、システム部が,P社開発成果物のレビュー・承認、C社利用部門との仕様などの調整を担当している」とあります。
また、「PMへのヒアリング」(2)では、「システムテストも**引き続きP社の協力を得て実施する予定**であり、懸念点はないと考えている」とPMOが回答しています。
システムテストやユーザ受入テストは、システムの**最終的な品質**を保証し、**ユーザーが利用可能であること**を確認する極めて重要なフェーズです。これを外部委託先（P社）任せにする体制では、C社側の責任や主体性が希薄になり、プロジェクトの成功に支障をきたすリスクがあります。C社システム部と利用部門が主体的に関与し、P社はあくまで支援という形で、C社が責任を持ってテストを実施する体制が適切かどうかを確認する必要があります。

> **正解**: C 社システム部及び利用部門が主体的にシステムテストを実施する体制を確保できるか

**思考プロセス**:
1.  〔追加の確認事項と検討事項〕(5)と「工程計画」の「システムテストにおいて、業務システム全てを連携させたテストとユーザ受入テストを行う計画」という記述に着目します。
2.  「開発体制」を読むと、P社が開発の中心であり、C社システム部はレビュー・承認、調整が主な役割であることが分かります。また、「PMへのヒアリング」(2)でPMOが「システムテストも引き続きP社の協力を得て実施する予定」と述べており、P社主導が続くことが示唆されています。
3.  システムテストは、システムが要件を完全に満たし、業務に適用可能であるかをC社自身が最終的に確認するフェーズです。これを外部委託先に「任せきり」にする体制は問題です。C社自身（システム部と利用部門）が**主体性**を持ってテストに臨むべきです。
4.  したがって、C社が主体的にシステムテストを実施できる体制が確保されているかを確認すべきであると判断できます。これを45字以内でまとめます。

**関連する重要な技術知識・用語の補足説明**:
*   **システムテスト**: 開発されたシステムが、機能要件・非機能要件を全て満たしているかを検証するテスト。通常は開発側（C社の場合はP社）が実施しますが、発注側（C社）の主体的な関与と承認が不可欠です。
*   **ユーザ受入テスト (UAT: User Acceptance Test)**: ユーザーが、実際に業務で使用できるレベルの品質や操作性、機能が実現されているかを確認するテスト。ユーザー部門が主体となって実施すべきテストです。
*   **プロジェクト体制と責任分担**: 外部委託契約においても、発注側はプロジェクトの最終的な品質や成功に対する責任を負います。特に最終フェーズのテストでは、発注側が主体的に関与し、成果物を検証する体制が重要です。
*   採点講評でも「システム開発を外部に委託している場合の結合テスト完了評価に対する監査の観点,運用すべき監査手続の知識と立案能力を問う」とあり、外部委託プロジェクトにおける発注側の責任と体制の確認の重要性が強調されています。

## 採点講評からのアドバイス
問3全体として、**外部委託されたシステム開発プロジェクト**における**結合テスト完了評価**の監査能力が問われています。特に、テストの「量」だけでなく「質」を評価する視点、具体的な課題（マスタデータ作成ミス）に対する**再発防止策**の確認、**システム間連携**が関連する課題における**部門横断的な情報共有**の重要性、そして最終的なシステムテストにおける**発注者側の主体的な関与**の必要性を理解することが重要です。
問題文中の具体的な事例を正確に読み取り、そこからリスクを抽出し、適切な監査手続や改善提案を導き出す実践的なスキルを磨きましょう。特に、**「なぜそれが問題なのか」「なぜそのような確認が必要なのか」**という根拠を論理的に説明できるようにすることが、解答力を高める上で役立ちます。
