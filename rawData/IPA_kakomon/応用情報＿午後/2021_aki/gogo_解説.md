# 問3 解説

## 問題の概要
本問題は、**グラフ理論**における**一筆書き**（Eulerian circuit）をテーマに、その経路を探索するアルゴリズムと、それを実装したプログラムの読解能力を問うものです。与えられた有向グラフにおいて、全ての辺を1回ずつ通り、出発点に戻る閉じた経路を見つけるための手順が説明され、その一部を関数`directedE`としてプログラムが示されます。

## 出題のポイント
この問題では、以下の知識とスキルが問われます。
-   **グラフ理論の基礎**: 有向グラフ、辺、点、経路、強連結グラフといった用語の理解。
-   **アルゴリズムの理解**: 一筆書き経路を探索する特定の手順（深さ優先探索に近い考え方）の理解。特に、探索済みの経路と未探索の辺、確定済みの経路という概念。
-   **データ構造**: グラフを表現するための配列（`edgefirst`, `edgenext`, `current`, `searched`, `path`）の役割と使い方。
-   **プログラミングの読解力**: 提示された擬似コード（関数`directedE`）の動作を正確にトレースし、変数や配列の値の変化、ループ、条件分岐を理解する能力。
-   **プログラミングにおける基本的な考え方**: ループ条件の境界値、無限ループの発生条件など。

### 設問1
図4中の「ア」～「エ」に入れる適切な字句を答える問題です。関数`directedE`のプログラムの穴埋めです。

#### 設問1 (ア)
**プログラム部分**:
```
while (last が1以上)
if (current [x] が ア でない)
```
> **正解**: 0

**思考プロセスと解説**:
配列`current[n]`は、「点nを始点とする未探索の辺の中で最小の番号を格納する。点nを始点とする未探索の辺がない場合は0を格納する。」と表2に定義されています。
`if (current[x] が ア でない)`という条件は、「点`x`からたどれる未探索の辺が存在する場合」を意味します。つまり、`current[x]`が0でない場合、点`x`から次に進むべき未探索の辺があるということです。
したがって、「ア」には未探索の辺がないことを示す値**0**が入ります。

#### 設問1 (イ)
**プログラム部分**:
```
searched [top] ← temp
current[x] ← イ
x ← end[temp]
```
> **正解**: edgenext[temp]

**思考プロセスと解説**:
この部分は、点`x`から`temp`（現在の探索辺）をたどった後、**次にたどるべき未探索の辺**を更新する処理です。
「次にたどるべき未探索の辺」は、現在の辺`temp`の次の接続辺、つまり`edgenext[temp]`で示されます。
配列`edgenext[m]`は、「辺mの次の接続辺の番号が格納されている。次の接続辺がない場合は0が格納されている。」と表2に定義されています。
したがって、「イ」には**edgenext[temp]**が入ります。

#### 設問1 (ウ)
**プログラム部分**:
```
else
top ← ウ
temp ← searched [top]
```
> **正解**: top - 1

**思考プロセスと解説**:
`else`節は、`if (current[x] が 0 でない)`の条件が偽、つまり「点`x`からたどれる未探索の辺がない場合」に実行されます。このとき、アルゴリズムの「一筆書きの経路の求め方」の(3)に従い、「探索済みの経路を未探索の辺が接続する点又は一筆書きの経路の出発点まで遡る。」必要があります。
配列`searched`は「一筆書きの経路を構成する探索済の辺の番号を順番に格納する。(探索済の経路)」と定義されており、`top`はその格納位置（次の格納先）を示しています。
遡る処理では、最後に探索した辺を取り除き（または確定し）、その前の点に戻ります。したがって、`searched`配列から一つ前の辺に戻るためには、`top`の値を1減らす必要があります。
よって、「ウ」には**top - 1**が入ります。

#### 設問1 (エ)
**プログラム部分**:
```
path[last] ← temp
x ← エ
last ← last - 1
```
> **正解**: start[temp]

**思考プロセスと解説**:
`else`節の中では、`searched[top]`が`temp`として取り出され、`path[last]`に格納されて確定済みの経路となります。
そして、`last`が1減らされ、次に確定する辺の位置を示します。
`x ← エ`は、「遡った辺 temp を確定済にする」処理の後、「未探索の辺が接続する点がある場合は、それを新たな出発点として、(2)に戻って新たな経路を見つける」というアルゴリズムの(3)の後半部分に対応します。
`temp`は今確定した辺の番号であり、この辺を確定した後、さらに遡るために、この辺の**始点**に戻って次の探索を開始します。
配列`start[m]`は、「辺mの始点の番号が格納されている。」と表2に定義されています。
したがって、「エ」には**start[temp]**が入ります。

### 設問2
図1のグラフで関数`directedE`を動作させたとき, `while`文中の`if`文は、何回実行されるか、数値を答える問題です。

#### 設問2
> **正解**: 16

**思考プロセスと解説**:
`while`文中の`if`文は、プログラムの主要な探索ロジックです。この`if`文が実行されるのは、グラフの辺を**探索する**とき（`current[x]`が0でない場合）と、**遡る**とき（`current[x]`が0の場合）の両方です。
「一筆書きの経路の求め方」の手順と図2の例を参考にします。
1.  **探索済の経路に追加されるとき**: 各辺は、出発点から探索されて`searched`配列に追加されるときに1回。
2.  **確定済の経路になるとき**: 遡る過程で、`searched`配列から`path`配列に移動するときに1回。

つまり、各辺は探索されて`searched`に入り、後に`path`に確定されるという2つの段階を経ます。プログラムの`if`文が1回実行されると`searched`に1つ辺が追加されるか、`path`に1つ辺が移動するかが起こります。
図1のグラフには**8つの辺**があります（辺1から辺8）。
それぞれの辺が`searched`に追加されるときに1回、`path`に移動して確定されるときに1回、合計2回処理されると考えられます。
したがって、`if`文の実行回数は、$8 \text{ (辺の数)} \times 2 = 16$回となります。

**採点講評からのアドバイス**:
採点講評では「設問2は,正答率が低かった。図2及び〔一筆書きの経路を求める手順〕から,グラフの各辺は,一度,探索済になってから確定済になることに注目して, if 文の実行回数が辺の数の2倍になることを理解してほしい」と指摘されています。このアルゴリズムの核心部分を理解しているかが問われる重要なポイントです。

### 設問3
一筆書きができない強連結な有向グラフで関数`directedE`を動作させたとき,探索はどのようになるかを、解答群の中から選び、記号で答える問題です。

#### 設問3
> **正解**: ウ (探索が無限ループに陥り、探索が終了しない。)

**思考プロセスと解説**:
一筆書きができるグラフの条件は、**全ての点において入次数と出次数が等しい**ことです（有向グラフの場合）。もし、この条件を満たさないグラフ（一筆書きができないグラフ）でこのアルゴリズムを実行した場合を考えます。

このアルゴリズムは、基本的に深さ優先探索のように辺をたどっていきます。
-   `if`節が実行される場合: 未探索の辺をたどって先に進みます。`top`は増え、`x`は移動します。
-   `else`節が実行される場合: 未探索の辺がないので、探索済みの辺を`path`に確定し、`top`は減り、`x`は**その辺の始点**に戻ります。

もし、ある点から未探索の辺が尽きて`else`節に入り、遡っても、一筆書きの条件を満たさないグラフでは、全ての辺をたどることができません。特に、**特定の点に出次数のみが多く残っている、または入次数のみが多く残っている**ような状況が発生すると、`current[x]`が常に0でない（つまり、永遠に新しい辺をたどり続ける）か、あるいは`current[x]`が常に0で`top`がどんどん減っていき（最終的に`top`が0以下になる）プログラムが異常終了する可能性があります。
しかし、このプログラムは`while (last が1以上)`という条件でループしています。`last`は最終的に`M`（辺の総数）から1まで減っていきます。
もし、一筆書きができないグラフで`searched`配列に登録された辺が`path`に移動しきらないうちに、すべての辺を探索できなくなり、`top`が減り続けると、最終的に`top`が1になった状態で`else`に入り、`top`を`top-1`すると`top`は0になり、`searched[0]`を参照しようとしてエラーになるか、あるいは`while`条件で`last`が1になったときに`x`が`start[temp]`に戻り、その点から再びループしてしまう可能性があります。
問題文中のアルゴリズムは、「出発点から辺の向きに従って辺を順番にたどり、出発点に戻る経路を見つける探索を行う」とあり、**必ず出発点に戻る**ことを前提としています。強連結な有向グラフであれば、どの点から出発しても必ず出発点に戻ることができます。
しかし、全ての辺を1度ずつ通るという要件を満たせない場合、アルゴリズムは同じ辺を繰り返し探索しようとしたり、ある部分グラフ内で無限ループに陥ったりする可能性があります。特に、`current[x]`が0にならない限り`top`が増え続け、`searched`に同じ辺が何度も追加される、あるいは`current[x]`が0になったときに`top`を減らして戻るものの、新たな辺を見つけられずに再び同じ経路をたどってしまう、といった状況が発生し、**無限ループ**に陥る可能性が高いです。
解答群のアとイは「終了する」とありますが、全ての辺を一筆書きできない場合、このアルゴリズムが正常終了することは期待できません。

### 設問4
図4のプログラムは、配列`searched`を配列`path`に置き換えることで、使用する領域を減らすことができる。このとき、無駄な繰返しが発生しないように、下線①の繰返し条件を、変数`top`と`last`を用いて変更せよ。
**下線①**: `while (last が1以上)`

> **正解**: top が last 以下

**思考プロセスと解説**:
現在のプログラムでは、`searched`と`path`という2つの配列が使われています。`searched`は探索中の経路、`path`は確定した経路を格納します。
`searched`を`path`に置き換えるということは、1つの配列`path`の中に、探索中の辺と確定済みの辺の両方を格納するように変更するということです。

アルゴリズムの動作を振り返ると：
-   `if`節では、新たな辺`temp`を`searched[top]`に格納し、`top`をインクリメントします。
-   `else`節では、`searched[top]`から`temp`を取り出し、`path[last]`に格納し、`top`と`last`をデクリメントします。

もし`searched`と`path`を同じ配列`path`で兼ねる場合：
-   `path`配列のインデックス`1`から`last`までは**確定済みの経路**が格納されます。
-   `path`配列のインデックス`last+1`から`top-1`までは**探索中の経路**が格納されます。

つまり、`path`配列は次のように構成されます：
`[確定済みの経路の辺 (1..last)] [探索中の経路の辺 (last+1..top-1)]`

`while`ループは、全ての辺が`path`に確定されるまで、つまり`last`が`M`に達するまで繰り返す必要があります。
現在の`while (last が1以上)`という条件では、`last`が1になるまでしか繰り返しません。しかし、`top`は探索中の辺の数を、`last`は確定済みの辺の数をそれぞれ管理しています。
プログラムが終了するのは、全ての辺が確定済みの経路に含まれたときです。このとき、`path`配列の全ての要素（$M$個の辺）が埋まり、`last`は$M$になります。
また、`top`は常に`last`以上である必要があります（`top`が指す位置が`last`が指す位置より前になることはない）。
探索中に`top`が増えて`path`に辺が追加され、遡る際に`top`が減り、確定されると`last`が減ります。最終的に全ての辺が確定されると、`top`と`last`は同じ値（$M+1$または$1$）になるはずです。
問題文には「全ての辺が確定済の経路になった時点で探索が完了して、その確定済の経路が一筆書きの経路になる」とあります。これは、`last`が`M`（辺の総数）になり、同時に`top`も`M+1`になる状態を示します。

新しい繰返し条件は、「まだ全ての辺が探索されていない、あるいは確定されていない状態」を表現する必要があります。
`top`は現在探索中の経路の次に辺を格納する位置を、`last`は確定済み経路の最後に辺を格納する位置を管理しています。
`top`が`last`以下になった場合、探索中の経路が空になるか、あるいは`path`の先頭要素を指すことになるため、これ以上探索が進まない状況、すなわち、これ以上経路を遡れないか、探索できる辺がないことを示します。

アルゴリズムの終了条件は「全ての辺が確定済の経路になった時点」です。プログラム中で`last`が`M`になることを目指す。
しかし、`while`文の現在の条件は`last`が`1`以上。
`if (current[x] が0でない)`のケースでは`top`がインクリメントされ、`else`ケースでは`top`と`last`がデクリメントされます。
探索が続いている間は、`top`は`last`よりも大きい位置を指しているはずです（`top`が次に辺を格納する位置、`last`が確定済みの経路の最後）。
例えば、図2[1]の時点では`searched`に4つの辺があり`top=5`、`path`は空で`last=8`。
図2[2]の時点では`searched`に3つの辺があり`top=4`、`path`に辺4が確定され`last=7`。
`top`が指す位置は常に`last`より大きいか等しいはず。
`top`は`searched`に辺が格納されるたびに1ずつ増えていく変数であり、`last`は`path`に辺が格納されるたびに1ずつ減っていく変数です。
全ての辺が`path`に移動した状態では、`top`は最終的に`last`と同じか、`last+1`（次の格納位置）になるはずです。
問題文の「下線①の繰返し条件を、変数 topとlast を用いて変更せよ」に注目します。
`searched`配列を`path`配列に置き換えるということは、探索中の辺も`path`に格納することになります。
このとき、`path[1]`から`path[last_actual]`までが確定済み、`path[last_actual+1]`から`path[top_actual-1]`までが探索中、というような使い方になるでしょう。
このアルゴリズムは、`top`と`last`という2つのポインタで、探索中の経路と確定済みの経路の境界を管理しています。
`top`は探索中の経路の**終端（次の書き込み位置）** を示し、`last`は確定済みの経路の**終端（現在の最後の要素位置）** を示します。
ループが続く条件は、まだ探索すべき辺があるか、確定すべき辺があるか、です。
これは、**まだ確定していない辺が`path`配列内に残っている（探索中の経路がある）**、あるいは**まだ全ての辺を確定しきっていない**、と解釈できます。
`top`は探索済み経路の次の要素位置を指し、`last`は確定済み経路の最後の要素位置を指します。
ループが続くのは、**探索中の経路に要素が残っている**場合、つまり`top`が`last`よりも大きい場合です。
最終的に、すべての辺が確定されると、`top`と`last`は同じ値（例えば、すべての辺が確定し終えた後の`top`は`M+1`、`last`は`0`から始まって`M`まで減っていくならば`1`になるまで、もし`M`から`1`まで減っていくならば`0`になるまで）になります。
プログラムの初期値を見ると、`top = 1`、`last = M`です。
`top`はインクリメントされていき、`last`はデクリメントされていきます。
ループ終了条件は、すべての辺が`path`に移動しきったとき、または探索できる辺がなくなったときです。
`top`と`last`が入れ替わる瞬間、または`top`が`last`より小さくなる瞬間に注目します。
`if`節が実行されるたびに`top`は増加し、`else`節が実行されるたびに`top`は減少します。
`last`は`else`節が実行されるたびに減少します。
`while`ループの継続条件は、**`top`が`last`以下ではないこと**、つまり**`top`が`last`よりも大きいこと**です。
`searched`配列を`path`配列に置き換える際、`searched`の要素は`path`の後半部分に格納され、`path`の要素は`path`の前半部分に格納されると考えると、`top`は`searched`の要素数、`last`は`path`の確定済みの要素数とみなせます。
`top`は、`path`配列の**探索中部分の次の書き込み位置**を指し、`last`は**確定済部分の現在の最後の要素位置**を指すとします。
例えば、`path`の容量が$M$である場合、`path[1..last]`が確定済み、`path[last+1..top-1]`が探索中とすると、`while`ループの条件は、`top`が`last`より大きい間、つまりまだ探索中の経路に辺が残っている間となります。

`top`の初期値は1、`last`の初期値はM。
`searched[top] ← temp` で `top` が増え、`path[last] ← temp` で `last` が減る。
この2つの配列を1つに統合する場合、例えば`path`配列のインデックスを`0`から`M-1`とすると、
`path[0...k]` が確定済み、`path[k+1...j]` が探索中、という管理方法になる。
この場合、`j`が`top`、`k`が`last`に対応する。
`top`が`last`よりも大きければ、まだ探索中の経路が存在し、ループを続ける。
`top`が`last`以下になれば、探索中の経路がなくなり、ループを終了する。
この考え方に従うと、`top が last 以下`という条件が適切です。

### 設問2

#### 設問2 (1)
本文中の下線①に該当するUTPケーブルの規格を、解答群の中から全て選び、記号で答える問題です。
**下線①**: 「フロア内の L2SW から DPC までの配線は、①1000BASE-T方式に対応した UTP ケーブルとした。」

> **正解**: イ, ウ, エ

**思考プロセスと解説**:
「1000BASE-T」は、**ギガビットイーサネット**のUTPケーブル規格です。
-   **カテゴリ3**: 10BASE-T（10Mbps）まで。
-   **カテゴリ5e**: 1000BASE-T（1Gbps）に対応した最低限の規格。
-   **カテゴリ6**: 1000BASE-Tに対応し、カテゴリ5eよりも高性能。
-   **カテゴリ6a**: 10GBASE-T（10Gbps）にも対応可能な高性能規格ですが、1000BASE-Tもサポートします。

したがって、1000BASE-Tに対応するのは**カテゴリ5e、カテゴリ6、カテゴリ6a**です。

-   **ファントムリード (Phantom Read)**: あるトランザクションが、条件に合致するレコードの集合を検索した後、別のトランザクションがその条件に合致する新しいレコードを挿入し、再度検索するとレコード数が増えている現象。

### 設問1

#### 設問1

> **正解**: プロセス

**思考プロセスと解説**:
問題文には「データ中心設計は、データの構造は機能と比較して変わりにくいという点に注目し、機能の設計に先行してデータの設計を行う手法である。」とあります。

これは、システムの処理手順（プロセス）を中心に設計を進める手法であり、**プロセス中心設計**と呼ばれます。
プロセス中心設計では、機能変更があると関連するデータも変更する必要が生じやすく、システムが肥大化・複雑化しやすいという課題があります。

**関連する重要な技術知識**:
-   **プロセス中心設計**: システムの機能や処理手順（プロセス）を起点に設計を進める手法。構造化設計などが含まれます。
-   **データ中心設計**: データの構造や関係性を起点に設計を進める手法。データの安定性に着目し、機能変更の影響を受けにくいシステムを目指します。
-   **オブジェクト指向設計**: データとプロセス（メソッド）を一体化したオブジェクトを中心に設計を進める手法。再利用性や変更容易性が高いとされます。

### 設問2
〔概念データモデルの設計〕について、(1)～(3)に答える問題です。

#### 設問2 (1)
図1中の「b」に入れる適切な字句を〔現行システム機能の調査〕内の字句を使って答える問題です。
**図1中の「b」**: 顧客エンティティのサブタイプ

**思考プロセスと解説**:
図1の概念データモデル（抜粋）では、`顧客`エンティティが`個人顧客`エンティティをサブタイプとして持っています。`個人顧客`はさらに`一般個人顧客`と`会員個人顧客`をサブタイプとしています。
〔現行システム機能の調査〕(1)顧客管理機能の項目には、「顧客には、顧客種別として、**個人顧客**と**法人顧客**があり、個人顧客には一般個人顧客とX社電子マネーをもつ会員個人顧客がある。」とあります。
この記述と図1の構造から、`個人顧客`と並ぶもう一つの顧客種別として、`顧客`エンティティのサブタイプには**法人顧客**が入るべきです。

**関連する重要な技術知識**:
-   **サブタイプ/スーパータイプ**: E-R図において、一般的なエンティティ（スーパータイプ）が、特定の特性を持つ複数の特殊なエンティティ（サブタイプ）に分類される関係。継承関係に似ています。

#### 設問2 (2)
図1中の通常商品を始点とし通常商品を終点とする1対多の関連は何を意味するか〔現行システム機能の調査〕内の字句を使って答える問題です。
**図1中の関連**: 通常商品 → 商品 (1対多)

> **正解**: 通常商品を束ねたセット商品

**思考プロセスと解説**:
図1では、`通常商品`から`商品`へ1対多の関連があります。
〔現行システム機能の調査〕(2)商品表示機能の項目に「商品には、商品種別として、**通常商品**のほか,通常商品を束ねた**セット商品**,特売商品,タイムセール商品、事前に予約することによって通常商品を割引価格で購入できる事前予約商品,及び顧客の購入履歴から算出したお勧め商品がある。」とあります。
「通常商品を束ねたセット商品」という記述が、`通常商品`が複数集まって`商品`の1種である`セット商品`を構成する関係を示しています。E-R図で1対多の関係は、親エンティティのインスタンス1つに対して、子エンティティのインスタンスが複数関連付けられることを示します。ここでは、1つの**セット商品**が複数の**通常商品**から構成される、と解釈できます。図の向きは逆ですが、セット商品が複数の通常商品を持つという意味で、1対多の「多」の側が通常商品です。
したがって、`通常商品`から`商品`への1対多の関連は、**通常商品を束ねたセット商品**を意味します。

#### 設問2 (3)
本文中の「c」「d」に入れる適切な字句を、解答群の中から選び、記号で答える問題です。
**本文中の「c」**: 「この概念データモデルのうち,通常商品と事前予約商品は c 関係,」
**本文中の「d」**: 「通常商品とお勧め商品は d 関係である。」

> **正解 (c)**: ウ (包含)
> **正解 (d)**: ア (共存)

**思考プロセスと解説**:
-   **通常商品と事前予約商品 (c)**: 図1の`商品`エンティティの下に`通常商品`、`事前予約商品`、`特売商品`、`タイムセール商品`、`お勧め商品`がサブタイプとして列挙されています。これらは全て`商品`の一種であり、**商品というスーパータイプの特定の分類**として扱われます。
    -   「包含関係」とは、あるエンティティが別のエンティティの特定のケースや分類である場合に使われます。`通常商品`と`事前予約商品`はどちらも`商品`エンティティのサブタイプであり、**商品という大きなカテゴリに包含される**関係です。これらは互いに排他的な関係ではなく、同じ商品が通常商品であると同時に事前予約商品になることはありませんが、商品というスーパータイプがこれらを「包含」しています。
    -   解答群のア共存、イ排他はサブタイプ同士の関係で、ウ包含はスーパータイプとサブタイプの関係で使われます。
    -   ここでいう「包含関係」は、一般的なオブジェクト指向の包含（あるオブジェクトが別のオブジェクトを部品として持つ）とは少し異なります。E-R図のサブタイプ関係を指しています。

-   **通常商品とお勧め商品 (d)**: `通常商品`と`お勧め商品`は、いずれも`商品`エンティティのサブタイプです。
    -   〔現行システム機能の調査〕(2)商品表示機能には「商品には、商品種別として、通常商品のほか, ... お勧め商品がある」とあります。
    -   `通常商品`は一般的な商品であり、`お勧め商品`は「顧客の購入履歴から算出された」商品です。
    -   1つの商品が`通常商品`であると同時に`お勧め商品`であることはありえます。例えば、ある通常の商品が、顧客の購入履歴に基づいて「お勧め商品」として表示されることもあります。これらは互いに排他的ではなく、**同じ商品が複数の属性を持つ**、または**複数の分類に属する**という意味で、**共存**しうる関係です。

**関連する重要な技術知識**:
-   **排他関係**: あるエンティティが複数のサブタイプを持つ場合、同時に複数のサブタイプに属することができない関係（例：男性と女性）。
-   **包含関係**: あるエンティティが複数のサブタイプを持つ場合、同時に複数のサブタイプに属することができる関係。または、スーパータイプがサブタイプを包含する関係。
-   **共存関係**: 複数の分類や特性が、一つのエンティティに対して同時に存在しうる関係。

### 設問3
〔顧客管理機能の設計〕について、(1), (2)に答える問題です。

#### 設問3 (1)
本文中の下線①について、一般個人顧客と会員個人顧客を二つのテーブルに分けるのではなく個人顧客というテーブルとした理由として、ふさわしくないものを解答群の中から選び、記号で答える問題です。
**下線①**: 「Y 君は、顧客管理機能の論理テーブルとして,①顧客種別,顧客,個人顧客,法人顧客の四つのテーブルを設計した。」

> **正解**: ウ (テーブルへの列追加時に顧客管理機能のソフトウェアの影響調査の範囲が小さくなるから)

**思考プロセスと解説**:
「一般個人顧客と会員個人顧客を二つのテーブルに分けるのではなく個人顧客というテーブルとした」のは、サブタイプを**スーパータイプに統合**する設計判断です。
-   **ア (一般個人顧客と会員個人顧客で属性に大きな差がないから)**: 属性に大きな差がない場合、テーブルを分けると冗長になるため、統合する理由は適切です。
-   **イ (顧客種別には、多くの変更が入らないことが予想されるから)**: 顧客種別の変更が少ない場合、テーブル統合による柔軟性低下の影響が小さいため、統合する理由は適切です。
-   **エ (販売実績の集計などを行う場合に、二つのテーブルではテーブル結合が多くなり、データベースサーバの負荷が大きいから)**: テーブルを分けるとJOIN操作が増え、性能低下を招く可能性があるため、統合する理由は適切です（性能重視の設計）。
-   **ウ (テーブルへの列追加時に顧客管理機能のソフトウェアの影響調査の範囲が小さくなるから)**: テーブルを統合した場合、新しい列を追加すると、そのテーブルを参照する**すべてのソフトウェア**に影響が及ぶ可能性があります。逆にテーブルを分けておけば、特定のサブタイプに固有の列追加であれば、そのサブタイプのテーブルを参照するソフトウェアのみに影響が限定されることが多いです。したがって、統合することで影響調査範囲が**大きくなる**可能性があり、この理由は**不適切**です。

**採点講評からのアドバイス**:
採点講評では「設問3(1)は,正答率が低かった。システム機能の設計において,システム性能と変更容易性はトレードオフの関係になる場合が多く,システムへのアクセス数や機能変更の頻度などを考慮した設計を行う必要がある。ここでは,システム性能に重点を置くことを考慮して正答を導いてほしい」と指摘されています。データベース設計における「性能」と「変更容易性」のトレードオフを理解し、どちらを重視した設計判断かを見極めることが重要です。

#### 設問3 (2)
本文中の下線②について、顧客クラスを定義することでソフトウェアの肥大化が防止できるのはなぜか,30字以内で述べよ。
**下線②**: 「ソフトウェアの肥大化を防止するために顧客クラスを定義し、顧客クラスを継承するクラスとして一般個人顧客,会員個人顧客,法人顧客の三つのクラスを設計した。」

> **正解**: 顧客種別によらない共通な処理を一か所で定義できるから。

**思考プロセスと解説**:
オブジェクト指向設計において、スーパークラス（親クラス）を定義し、サブクラス（子クラス）がそれを継承する**継承関係**は、**コードの再利用性**を高め、**共通処理の集約**を可能にします。
`顧客`クラスというスーパークラスを定義することで、`一般個人顧客`、`会員個人顧客`、`法人顧客`といったサブクラスに共通する属性（例：顧客番号、配送先住所）やメソッド（例：顧客番号を取得する`get顧客番号()`）を、`顧客`クラスにまとめて定義できます。
これにより、各サブクラスで同じコードを重複して書く必要がなくなり、コード量が削減され、結果的に**ソフトウェアの肥大化が防止**されます。また、共通処理の変更があった場合でも、スーパークラスだけを修正すれば済むため、変更容易性も向上します。

## 設問4
〔商品表示機能の設計〕について、(1), (2)に答える問題です。

#### 設問4 (1)
本文中の下線③の設計とすることで、商品種別を追加した際に、運用環境へのリリース時にどのような作業を低減できるか,20字以内で述べよ。
**下線③**: 「商品表示機能の論理テーブルとして、③特売商品テーブル、セット商品テーブルなど商品種別ごとに多数のテーブルを作成するのではなく、商品種別と商品の二つのテーブルを作成し、運用環境へのリリース時の作業量を低減する設計とした。」

> **正解**: データベースへのテーブル追加作業

**思考プロセスと解説**:
「商品種別ごとに多数のテーブルを作成する」設計（例：`特売商品`テーブル、`セット商品`テーブルなど）では、新しい商品種別が追加されるたびに、**新しいテーブルをデータベースに追加**する必要があります。このテーブル追加作業は、運用環境でのデータベーススキーマ変更を伴い、通常、慎重な計画とテストが必要な、時間と手間のかかる作業です。
しかし、「商品種別と商品の二つのテーブルを作成し」という設計では、商品種別自体は`商品種別`テーブルにレコードとして追加されるだけで済み、`商品`テーブルには新たな商品種別の商品が格納されます。この場合、データベーススキーマの変更は不要であり、**データベースへのテーブル追加作業**が大幅に低減されます。

**採点講評からのアドバイス**:
採点講評では「設問4(1)は,正答率が低かった。機能変更に伴うテーブル追加作業のリスクについて問うた。ソースコードの変更はテスト環境で十分なテストができるが、テーブル追加作業はリリース時の限られた時間内で作業及びテストを行う必要があり,リスクが高い。システム設計においては,システム性能や変更容易性への考慮にとどまらず、リリース作業におけるリスクについても是非考慮してほしい」と指摘されています。システム設計の検討においては、運用・保守フェーズにおける作業コストやリスクも考慮することが重要です。

#### 設問4 (2)
本文中の下線④について、Y君が商品種別ごとにクラスを定義した理由を、商品表示機能の特徴の観点から20字以内で述べよ。
**下線④**: 「ソフトウェア設計としては商品クラスを定義するとともに、④商品種別ごとに個別のクラスを設計した。」

> **正解**: 画面の表示方法が異なるから。

**思考プロセスと解説**:
図3のソフトウェアのクラス図を見ると、`商品`クラスの下に`通常商品`、`特売商品`、`タイムセール商品`、`事前予約商品`、`セット商品`といった個別のクラスが定義されています。それぞれのクラスには`画面表示()`というメソッドがあります。
〔現行システム機能の調査〕(2)商品表示機能の項目には、「商品種別ごとに**画面の表示方法が異なる**。この機能は、顧客にX社のネットスーパーを選択してもらうための重要な機能であり、商品種別の追加に関する機能変更が多い。」とあります。
