### 設問3 (3)
本文中の下線④の変更後の案について、表6中のルール1に記述すべきパターンを、図5の記述形式で答えよ。

> **正解**: `\W[jJ][nN][dD][iI]\W` または `\W(j|J)(n|N)(d|D)(i|I)\W`

図5のWAFルールの記述形式では、正規表現のルールとして以下が示されています。
*   `[xyz]`: `x`, `y`, `z` のいずれか1文字にマッチする。

*   `n`または`N`: `[nN]` または `(n|N)`
*   `d`または`D`: `[dD]` または `(d|D)`
*   `i`または`I`: `[iI]` または `(i|I)`

**誤答しやすいポイントや注意点**:

**誤答しやすいポイントや注意点**:
「通信が遮断される」という結果だけを記述するのではなく、「正常な通信を異常として検知する」という検知の誤り自体が弊害であることを説明する必要があります。

### 設問3 (1)

**解答に至るための思考プロセス**:

設定Pは、「VPN-Hに、どのような通信要求に対しても応答しない“Deny-ALL”を設定した上で、あらかじめ設定されている**順番**にポートに通信要求した場合だけ所定のポートへの接続を許可するという設定」と説明されています。

この設定Pは、特定のポートへのアクセスが、事前に定義された**特定の順序**で一連の通信が行われた場合にのみ許可されるというものです。もし攻撃者が、正規のユーザーが行った一連の通信パケットを**盗聴**し、そのパケットの内容と**送信順序**を正確に把握できた場合、攻撃者はその盗聴したパケットを**同じ順序で再送信する**ことで、設定Pを突破し、ポートへの接続を許可させることが可能となります。これは**リプレイ攻撃**の一種です。

**関連する重要な技術知識・用語の補足説明**:
*   **ポートノッキング (Port Knocking)**: 特定のポートへの接続要求を、事前に定義された順序で複数回行うことで、本来閉じているポートを開放させる技術です。セキュリティを高めるために利用されますが、本件の設定Pのように単純な順序の検証だけでは、リプレイ攻撃に脆弱になる可能性があります。
*   **盗聴 (Eavesdropping)**: ネットワーク上を流れる通信内容を不正に傍受する行為です。
*   **リプレイ攻撃 (Replay Attack)**: 盗聴した正当な通信（認証情報や操作リクエストなど）を再度送信することで、認証を突破したり、不正な操作を行ったりする攻撃です。

**誤答しやすいポイントや注意点**:
設定Pの「順番」というキーワードの重要性を見落とさないように注意しましょう。単にパケットを盗聴するだけでなく、そのパケットを「同じ順番」で送信することが突破方法であるという点がポイントです。

### 設問3 (2)
本文中の下線④について、突破されないのはなぜか。40字以内で答えよ。

> **正解**: SPA パケットはユニークであり、同じパケットを再利用すると破棄されるから

**解答に至るための思考プロセス**:
問題文P18のT主任とL部長の会話で、「SPAなら、④攻撃者が何らかの方法でパケットを盗聴できたとしても、突破はされません。」とあります。
表6「SPAの主な仕様」の項番2には、「SPA パケットには**ランダムデータ**が含まれており、送信先で検証される。**以前受信したものと同じランダムデータをもつ SPA パケットを受信した場合は、破棄される**。」と説明されています。

このSPAの仕様により、SPAパケットは毎回異なるランダムデータ（nonceなど）を含むため、事実上**ユニーク（一意）** なものとなります。攻撃者がSPAパケットを盗聴したとしても、そのパケットをそのまま再利用して送信しようとすると、送信先（S-APPLなど）で「以前受信したものと同じランダムデータ」と判断され、リプレイ攻撃として**破棄されます**。これにより、盗聴されたSPAパケットが不正に再利用されることを防ぎ、SPAはリプレイ攻撃に対して高い耐性を持つことができます。

**関連する重要な技術知識・用語の補足説明**:
*   **SPA (Single Packet Authorization)**: 単一の暗号化された認証パケットを送信することで、ファイアウォールなどのアクセス制御デバイス越しに、特定のサービスポートを一時的に開放させるプロトコルです。ポートノッキングとは異なり、パケットの内容が暗号化され、ランダムなデータ（ノンス）を含むことでリプレイ攻撃に強く、より安全なアクセス制御を実現します。
*   **ノンス (Nonce)**: 「Number used once」の略で、一度だけ使用される使い捨てのランダムな数値や文字列です。リプレイ攻撃を防ぐために、認証プロトコルや暗号通信などで利用されます。

**誤答しやすいポイントや注意点**:
「ユニークである」だけでなく、「再利用されると破棄される」というSPAの具体的な動作まで含めて説明することで、リプレイ攻撃に対する耐性を明確にすることができます。

*   **S-APPLによる制限**: S-APPLの「接続サーバ許可機能」（表7項番4）により、VPN確立後に取引専用PCからアクセス可能なサーバーを取引先向けWebサーバーだけに限定します。

結果として、DDoS攻撃の多くを占める不特定多数からの攻撃や、ボットネットからの大量通信は、取引先向けWebサーバーに到達する前にS-APPLやUTMで遮断されるため、取引先向けWebサーバーがDDoS攻撃の直接的な影響を受けることを大幅に軽減できます。

**関連する重要な技術知識・用語の補足説明**:
*   **アクセス制御リスト (ACL)**: ネットワークデバイス（ファイアウォール、ルーターなど）で、特定のIPアドレス、ポート番号、プロトコルなどに基づいて通信を許可または拒否するルールです。
*   **ゼロトラスト (Zero Trust)**: 「決して信頼せず、常に検証せよ」という考え方に基づき、社内外の全てのアクセスを常に検証し、最小限の権限を与えるセキュリティモデルです。ここでは、取引先向けWebサーバーへのアクセスを厳格に制限することで、ゼロトラストの原則を部分的に適用しています。

**誤答しやすいポイントや注意点**:
S-APPLとUTMそれぞれの役割と、それらが連携することによるDDoS軽減効果を明確に記述することが重要です。「アクセスが制限される」だけでなく、「なぜDDoSが軽減されるのか」（不特定多数からの攻撃が到達しないため）まで説明しましょう。

# 問3 解説

## 問題の概要
D社がクラウドW上に新たに構築するECサイトXと商品企画サイトYに対して、セキュリティベンダーのZ社が**Webアプリケーション診断**を実施しました。この問題では、診断で検出された主要なWeb脆弱性である**クロスサイトスクリプティング (XSS)**、**クロスサイトリクエストフォージェリ (CSRF)**、**認可制御の不備**、および**サーバサイドリクエストフォージェリ (SSRF)**の**攻撃手法と具体的な対策**について検討するシナリオです。

## 出題のポイント
この問題は、Webアプリケーションセキュリティに関する広範な知識と、それぞれの脆弱性に対する攻撃手法および対策を深く理解しているかが問われます。具体的には、以下の知識・スキルが重要です。
*   **XSS (Cross-Site Scripting)**: 格納型XSSの攻撃手法、セッションハイジャックなどによる被害、およびサニタイジング/エスケープ処理による対策。
*   **CSRF (Cross-Site Request Forgery)**: 攻撃手法、CSRFトークンの役割と、その不適切な実装による脆弱性、Cookieの`SameSite`属性による対策。
*   **認可制御の不備 (IDOR)**: `order-code`のような直接参照オブジェクトを推測・改変する攻撃手法と、適切な認可チェックによる対策。
*   **SSRF (Server-Side Request Forgery)**: サーバーが外部URLにリクエストを送信する機能を悪用し、内部リソース（特に**IMDS**）にアクセスする攻撃手法、およびURLのホワイトリスト化による対策。
*   **IMDS (Instance Metadata Service)**: クラウド環境におけるインスタンスメタデータサービスからのクレデンシャル情報取得方法（方式1、方式2）と、SSRFによる悪用リスク。
*   **HTTPプロトコル**: GET/POSTメソッドの違い、HTTPヘッダ、Cookieの`SameSite`属性の動作。

*   **クロスサイトスクリプティング (XSS)**: Webページに悪意のあるスクリプトを埋め込み、そのページを閲覧したユーザーのブラウザ上で実行させる攻撃です。これにより、セッションクッキーの窃取、ユーザーの個人情報詐取、偽のコンテンツ表示などが可能になります。本件は、入力されたスクリプトがデータベースなどに保存され、別のページで表示される際に実行される**格納型XSS (Stored XSS)**に該当します。

**誤答しやすいポイントや注意点**:

    *   POSTリクエストの場合 (b): 外部からの遷移なので、Cookieは送信されません。したがって、**b: ×**

    *   POSTリクエストの場合 (d): POSTリクエストによるクロスサイトリクエストではCookieは送信されません。したがって、**d: ×**

*   **SameSite属性**: Cookieに設定することで、ブラウザがクロスサイトリクエスト時にCookieを送信するかどうかを制御するための属性です。CSRF対策に非常に有効です。
    *   `Strict`: 最も厳格で、CSRF対策として最も強力ですが、利便性を損なう場合があります。

### 設問2 (6)
図4中の `h` に入れる適切なソースコードを答えよ。

> **正解**: finally

**解答に至るための思考プロセス**:
図4のソースコードの28行目に`h`が空欄となっています。これは、`try-catch`ブロックの後に、メソッドの終了前に位置するブロックです。
問題文P37のE氏の指摘に、「利用する AP サーバの実装では、変数 `psObj` の指すメモリ領域においてメモリリークが発生する可能性がある。」とあります。
図4の`addUser`メソッドでは、`PreparedStatement psObj;`が宣言され、DB操作に利用されています。データベース接続や`PreparedStatement`などのDBリソースは、処理が完了したら必ずクローズして解放する必要があります。
Javaにおいて、例外の発生有無にかかわらず、**必ず実行される**処理を記述するブロックは**`finally`**ブロックです。
`finally`ブロック内に`psObj`のクローズ処理（`psObj.close()`）を記述することで、`try`ブロック内で例外が発生した場合でも、`catch`ブロックで例外が処理された後でも、リソースが確実に解放され、メモリリークを防ぐことができます。

**関連する重要な技術知識・用語の補足説明**:
*   **finallyブロック (Java)**: `try-catch`文の一部で、`try`ブロック内で例外が発生したかどうかに関わらず、常に実行されるコードブロックです。ファイルやデータベース接続などのリソースを確実に解放するために利用されます。
*   **リソースリーク (Resource Leak)**: プログラムが確保したリソース（メモリ、ファイルハンドル、データベース接続など）が、不要になった後も解放されずに残り続けてしまう状態です。システムのパフォーマンス低下や障害の原因となります。

**誤答しやすいポイントや注意点**:
`finally`ブロックは例外の有無にかかわらず実行されるという特性を理解しているかが問われます。`catch`ブロック内でリソースをクローズすると、`try`ブロックで例外が発生しなかった場合にクローズされない可能性があるため、`finally`ブロックの利用が推奨されます。

### 設問2 (7)
本文中の下線②について、図4の6,7行目をどのように修正すればよいか。修正後の適切なソースコードを解答群の中から選び、記号で答えよ。ここで、変数saltには, addUserメソッドの呼出しごとに異なる 32 バイトの固定長文字列が入っているものとし、ユーザーマスターテーブルの定義に変更はないものとする。

> **正解**: ア

**解答に至るための思考プロセス**:
問題文P37に、「E氏は、図4のソースコードでは、レインボーテーブル攻撃を受けたときに攻撃が成立してしまうので、図2の仕様及び②図4のソースコードの6,7行目を修正すべきであると指摘した。」とあります。
**レインボーテーブル攻撃**は、あらかじめ計算されたハッシュ値のリスト（レインボーテーブル）を用いて、パスワードのハッシュ値から元のパスワードを効率的に特定する攻撃です。
この攻撃を防ぐためには、パスワードのハッシュ化に**ソルト (salt)** を用いることが有効です。ソルトは、パスワードごとに異なるランダムな文字列であり、パスワードと結合してハッシュ化することで、同じパスワードでも異なるハッシュ値が生成され、レインボーテーブル攻撃や辞書攻撃に対する耐性を高めます。

設問の条件として「変数`salt`には, `addUser`メソッドの呼出しごとに異なる 32 バイトの固定長文字列が入っているものとし、ユーザーマスターテーブルの定義に変更はないものとする。」とあります。
図4の6,7行目は以下の通りです。
```java
6: byte[] hashByte = mdObj.digest(this.password.getBytes());
7: this.password = String.format("%x", new BigInteger(1, hashByte));
```

これをソルト付きハッシュに変更するためには、`this.password.getBytes()`の代わりに`salt`と`this.password`を結合したバイト列をハッシュ化する必要があります。
解答群を比較すると、**ア**が最も適切な実装です。
```java
ア: byte[] hashByte = mdObj.digest((salt + this.password).getBytes());
   this.password = salt + String.format("%x", new BigInteger(1, hashByte));
```
この修正では、`salt`と`this.password`を文字列として結合し、そのバイト配列をハッシュ化しています。そして、データベースに保存する`this.password`の値は、ソルトとハッシュ値を結合した形式にすることで、後でソルト値も利用してハッシュ値の検証が可能になります。

**関連する重要な技術知識・用語の補足説明**:
*   **ソルト (Salt)**: パスワードをハッシュ化する際に、個々のパスワードに対してランダムに生成・付加されるデータです。ソルトとパスワードを結合してハッシュ化することで、同じパスワードでも異なるハッシュ値が生成され、レインボーテーブル攻撃や辞書攻撃に対する耐性を高めます。
*   **鍵導出関数 (KDF)**: パスワードハッシュには、計算に時間がかかるように設計されたPBKDF2、bcrypt、scryptなどの鍵導出関数を用いるのが現代的なベストプラクティスです。ここでは汎用ハッシュ関数でソルトを適用する例が示されています。

**誤答しやすいポイントや注意点**:
ソルトをパスワードと結合してハッシュ化し、さらにDBに保存するハッシュ値にもソルトを含める（またはソルトとハッシュ値を別々に保存する）という一般的なソルトの利用方法を理解しているかが問われます。他の選択肢は、ソルトを正しく利用していなかったり、ハッシュ化の方式として適切でなかったりします。
