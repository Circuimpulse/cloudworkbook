### イ

**設問の内容**: SSL-VPNの方式として、リバースプロキシ方式、ポートフォワーディング方式、および空欄イ方式の3つが挙げられている。

**関連する重要な技術知識**:

### カ
> **正解**: DHE

**設問の内容**: TLS1.3で規定されている鍵交換方式は、空欄カ, ECDHE, PSKの3方式である。
**解答に至る思考プロセス**:
問題文には「TLS1.3 で規定されている鍵交換方式は、**カ**, ECDHE, PSKの3方式である。」とあります。TLS1.3で規定されている鍵交換方式は、**Diffie-Hellman (DH)**、**ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)**、および**PSK (Pre-Shared Key)**です。DHはDH鍵交換アルゴリズムを指し、DHEはEphemeral（一時的）なDH鍵交換を指します。文脈から「DHE」が適切です。
**関連する重要な技術知識**:
-   **Diffie-Hellman鍵交換 (DH)**: 公開鍵暗号の原理を利用して、安全でない通信路を通じて共通の秘密鍵を共有するためのアルゴリズム。
-   **ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)**: 楕円曲線暗号とDH鍵交換を組み合わせたもので、Forward Secrecy（前方秘匿性）を実現できます。セッションごとに一時的な鍵ペアを生成するため、過去の通信が将来の秘密鍵漏洩によって復号されるリスクを低減します。
-   **PSK (Pre-Shared Key)**: あらかじめ共有された秘密鍵を用いる方式。
-   **Forward Secrecy（前方秘匿性）**: 現在の通信セッションの秘密鍵が将来漏洩しても、過去の通信の内容が復号されることを防ぐ性質。TLS1.3では前方秘匿性を持つ鍵交換方式が必須とされています。

> **正解**: クライアント証明書の公開鍵に対する秘密鍵は本人しか保有していないから

**設問の内容**: 本文中の下線③「クライアント証明書で送信元の身元を一意に特定できる」理由を、「秘密鍵」という用語を用いて40字以内で述べよ。

**関連する重要な技術知識**:
-   **PKI（公開鍵基盤）**: 電子証明書と認証局（CA）を利用して、公開鍵の正当性を保証し、安全な通信を確立するための仕組み。

### (2)
> **正解**: CAのルート証明書

**設問の内容**: 本文中の下線④「クライアント証明書の検証のために、あらかじめSSL-VPN装置にインストールしておくべき情報」を答えよ。
**解答に至る思考プロセス**:
問題文には「TLS プロトコルのネゴシエーション中に、**④クライアント証明書が SSL-VPN 装置に送信され、SSL-VPN 装置で検証される**。」とあります。
クライアント証明書を検証するためには、その証明書を発行した**認証局（CA）の公開鍵**が必要です。特に、信頼の起点となる最上位の認証局（ルートCA）の証明書（**ルート証明書**）を事前に信頼するデバイス（ここではSSL-VPN装置）にインストールしておくことで、そのCAが発行した証明書を信頼できると判断します。
**関連する重要な技術知識**:
-   **ルート証明書**: 認証局の公開鍵が含まれる自己署名証明書で、信頼の連鎖の起点となります。OSやブラウザには主要なルート証明書があらかじめインストールされています。
-   **信頼の連鎖**: 証明書が有効であることを確認するために、発行元CAの証明書をたどっていくプロセス。最終的に信頼済みのルート証明書に行き着けば、その証明書は信頼できると判断されます。

### (3)
> **正解**: なりすまされたSSL-VPN装置へ接続してしまうリスク

問題文には「**⑤ SSL-VPN 装置からサーバ証明書が個人PCに送られ、個人PC で検証される**。」とあります。
サーバ証明書の検証は、クライアントが接続しようとしているサーバが、意図した正規のサーバであることを確認するために行われます。この検証が適切に行われない場合、悪意のあるサーバが正規のサーバになりすまして待ち受けている可能性があり、クライアントがそれに気づかずに接続してしまうリスクがあります。これを防ぐためにサーバ証明書の検証が必要です。

-   **サーバ認証**: クライアントが接続先のサーバが正規のものであることを確認するプロセス。中間者攻撃やフィッシングを防ぐために重要です。

### (4)
> **正解**: 秘密鍵が漏えいする前に行われた通信のデータ

**設問の内容**: 本文中の下線⑥について、TLS1.3で規定されている鍵交換方式に比べて、広く復号されてしまう通信の範囲に含まれるデータは何か。「秘密鍵」と「漏えい」という用語を用いて、25字以内で答えよ。
**解答に至る思考プロセス**:
問題文には「TLS プロトコルにおける鍵交換の方式には、クライアント側でランダムなプリマスタシークレットを生成して、サーバの RSA **工**鍵で暗号化してサーバに送付することで共通鍵の共有を実現する, RSA 鍵交換方式がある。また、Diffie-Hellman アルゴリズムを利用する鍵交換方式で, DH 公開鍵を静的に用いる方式もある。これらの方式は、**⑥秘密鍵が漏えいしてしまったときに不正に復号されてしまう通信のデータの範囲が大きい**という問題があり, TLS1.3 以降では利用できなくなっている。」とあります。
RSA鍵交換方式では、セッション鍵の元となるプリマスタシークレットがサーバの公開鍵で暗号化され、サーバの秘密鍵で復号されます。このため、一度サーバの秘密鍵が漏えいしてしまうと、その**秘密鍵が漏えいする前に行われた通信**であっても、そのセッションで利用された暗号化されたプリマスタシークレットが復号され、結果として過去の通信内容全体が復号されてしまう可能性があります。これを**前方秘匿性がない**と表現します。TLS1.3ではこの問題を解決するため、Forward Secrecyを提供する鍵交換方式（ECDHE, DHEなど）が必須とされました。
**誤答しやすいポイント**: 「漏えいした後の通信」ではなく、「漏えいする前の過去の通信」も復号されてしまう点がポイントです。

### (5)
> **正解**: 署名に用いる鍵: 利用者の秘密鍵
> 署名の検証に用いる鍵: 利用者の公開鍵

**設問の内容**: 本文中の下線⑦について、利用者がCAサービスにCSRを提出するときに署名に用いる鍵は何か。また、CAサービスがCSRの署名の検証に用いる鍵は何か。本文中の用語を用いてそれぞれ答えよ。
**解答に至る思考プロセス**:
問題文には「新しいクライアント証明書が必要なときは、利用者の公開鍵と秘密鍵を生成し、公開鍵から証明書署名要求(CSR)を作成して、CAサービスへ提出する。CA サービスは、クライアント証明書を発行してよいかどうかを K 社の管理者に確認するとともに、**⑦CSRの署名を検証して**、クライアント証明書を発行する。」とあります。
CSR (Certificate Signing Request) は、証明書の発行をCAに要求する際に、自身の**公開鍵**と識別情報を含んで作成されます。このCSRが間違いなく本人によって作成されたものであることをCAが確認するため、利用者は自身の**秘密鍵**でCSRに署名を付与します。CAサービスは、CSRに含まれる利用者の**公開鍵**を使って、その署名が正当なものであるかを検証します。
**関連する重要な技術知識**:
-   **CSR (Certificate Signing Request)**: 証明書署名要求。証明書を発行してもらう際にCAに送付するデータ。公開鍵、識別名、そしてこれらに対する所有者の秘密鍵による署名が含まれます。

- L3SW11 → M社 $\to$ L3SW31 ($20+50 = 70$) + L3SW31からサーバセグメント ($20$) = $90$
- L3SW11 → N社 $\to$ L3SW31 ($20+50 = 70$) + L3SW31からサーバセグメント ($20$) = $90$
- L3SW11 → M社 $\to$ L3SW32 ($20+50 = 70$) + L3SW32からサーバセグメント ($20$) = $90$
- L3SW11 → N社 $\to$ L3SW32 ($20+50 = 70$) + L3SW32からサーバセグメント ($20$) = $90$

やはり90になります。

**唯一、コスト70になる可能性**:
L3SW11から見たL3SW31/L3SW32へのコストが70であるという解釈。
「サーバセグメント」というより、「L3SW31/L3SW32」を中継地点として見た場合のコスト。
しかし設問は「サーバセグメントへの」経路コストを聞いているため、これは不適切。
これは非常に難しい問題です。採点講評からも難易度の高さが伺えます。
**最終的に解答例の「70」に合わせるための解釈**:
L3SW11から広域イーサネットへの出口コストが20。
広域イーサネット区間のコストが50（これは「リンク」のコストと読む）。
合計で $20+50 = 70$
この70は、L3SW11がL3SW31/32に到達するまでのコストです。
そして、L3SW31/32からサーバセグメントへはそれぞれコスト20で直結です。
もし、OSPFがL3SW31とL3SW32を「同一コストでサーバセグメントへ到達できるネクストホップ」と判断し、そのネクストホップへのコストを問うているとすれば、70はありえます。
そして、経路数はL3SW11からL3SW31への2経路 (M社経由, N社経由) と、L3SW11からL3SW32への2経路 (M社経由, N社経由) の合計4経路、これらがそれぞれネクストホップ（L3SW31またはL3SW32）に到達するためのコスト70となる。
この解釈が解答例に沿うものとなります。

**解答に至る思考プロセス（解答例に沿う解釈）**:
図4のL3SW11からデータセンタのL3SW31またはL3SW32への経路に着目します。
-   **L3SW11からL3SW31への経路**:
    -   M社広域イーサネット経由: L3SW11のM社側ポートコスト20 + M社広域イーサネットのリンクコスト50 = 70
    -   N社広域イーサネット経由: L3SW11のN社側ポートコスト20 + N社広域イーサネットのリンクコスト50 = 70
-   **L3SW11からL3SW32への経路**:
    -   M社広域イーサネット経由: L3SW11のM社側ポートコスト20 + M社広域イーサネットのリンクコスト50 = 70
    -   N社広域イーサネット経由: L3SW11のN社側ポートコスト20 + N社広域イーサネットのリンクコスト50 = 70
このように、L3SW11からサーバセグメントの直前のL3SW31/32までには、コストが70の経路が4つ存在します。サーバセグメント自体はL3SW31/32の直結ネットワークであり、L3SW31/32からサーバセグメントへのコストは20ですが、ECMPは「ネクストホップへの同ーコストの複数の経路」に対して適用されるため、L3SW11が認識するL3SW31/32へのコストを「70」と判断し、これらの4経路を等コスト経路として扱います。
**経路数**: L3SW11からL3SW31へ2経路、L3SW11からL3SW32へ2経路で、合計**4経路**。
**コスト**: 各経路のコストは**70**。
これは、L3SW11がサーバセグメントへの経路を計算する際に、L3SW31とL3SW32を等価なルータとして扱い、そのルータまでのコストを問うている、という特殊な解釈である可能性が高いです。

### (3)
> **正解**: パケット到着順序の逆転が起こりにくいから

**設問の内容**: 本文中の下線⑬について、フローモードの方が通信品質への影響が少ないと判断した理由を35字以内で述べよ。
**解答に至る思考プロセス**:
問題文には「K社で利用している L3SW のベンダに ECMP の経路選択の仕様を問い合わせたところ、次の仕様であることが分かった。…動作モードとして、パケットモードとフローモードがある。…パケットモードの場合、パケットごとにランダムに経路を選択し,フローモードの場合は、送信元 IPアドレスと宛先IPアドレスからハッシュ値を計算して経路選択を行う。P主任は、K社の社内のPCと業務サーバ間の通信における**⑬通信品質への影響**を考慮して、フローモードを選択することにした。」とあります。
**パケットモード**では、個々のパケットが異なる経路を通る可能性があるため、経路遅延の差によって**パケットの到着順序が逆転**するリスクが高まります。パケットの到着順序逆転は、TCP通信の再送処理を頻発させ、スループット低下や応答時間悪化など、通信品質に悪影響を及ぼします。
一方、**フローモード**では、同一フロー（通常は送信元IP/宛先IP/送信元ポート/宛先ポートの組）のパケットは常に同じ経路を通るため、パケットの到着順序逆転のリスクが低減され、通信品質への影響が少なくなります。
**関連する重要な技術知識**:
-   **ECMP (Equal Cost Multi-path)**: 等コストの複数の経路間でトラフィックを負荷分散する技術。
-   **フローベースルーティング**: トラフィックをフロー単位で識別し、同一フローは同一経路で転送する方式。パケットモードに比べて、通信の順序性を維持しやすい。

### (4)
> **正解**: 送信元 IPアドレスと宛先 IP アドレスから計算したハッシュ値が偏らないから

**設問の内容**: 本文中の下線⑭について、利用率がほぼ均等になると判断した理由をL3SWのECMPの経路選択の仕様に照らして、45字以内で述べよ。
**解答に至る思考プロセス**:
問題文には「P主任は、…フローモードを選択することにした。また、フローモードでも**⑭複数回線の利用率がほぼ均等になると判断した**。」とあります。ECMPの仕様として、「フローモードの場合は、送信元 IPアドレスと宛先IPアドレスからハッシュ値を計算して経路選択を行う。」と記述されています。
フローモードは、送信元IPアドレスと宛先IPアドレス（またはポート番号なども含めて）からハッシュ値を計算し、その結果に基づいて経路を選択します。利用率がほぼ均等になるためには、この**ハッシュ値が特定の経路に偏らず、複数の経路にバランス良く分散される**必要があります。もしハッシュ値の計算結果が偏ると、特定の経路ばかりが使われ、利用率にばらつきが生じてしまいます。
**関連する重要な技術知識**:
-   **ハッシュベース負荷分散**: 複数の経路がある場合に、パケットのヘッダ情報（IPアドレス、ポート番号など）からハッシュ値を計算し、その結果に基づいて転送経路を決定する方式。

### (5)
> **正解**: インタフェースの障害を検知した時に L3SW31 の VRRP の優先度を下げる。

**設問の内容**: 本文中の下線⑮について、この設定によるVRRPの動作を「優先度」という用語を用いて40字以内で述べよ。
**解答に至る思考プロセス**:
問題文には「P主任は、…サーバセグメントに接続されている L3SWの冗長化について、図5のように行うことにした。図5において、L3SW31 と L3SW32でVRRPを構成し、L3SW31 がVRRP マスタとなるように優先度を設定する。また、L3SW31 において、**⑮図5中のa又はbでの障害をトラッキングするようにVRRP の設定を行う**。これによって、a又はbのインタフェースでリンク障害が発生した場合でも、業務サーバからPCへのトラフィックの分散が損なわれないと考えた。」とあります。
VRRP (Virtual Router Redundancy Protocol) は、複数のルータで仮想ルータを構成し、その中で最も優先度の高いルータがマスタルータとなります。**障害トラッキング（トラック機能）**とは、ルータの特定の物理インターフェースやルートの稼働状況を監視し、障害を検知した場合に自身のVRRP優先度を自動的に下げる機能です。
この設定により、L3SW31のaまたはbのインターフェースに障害が発生した場合、L3SW31は自身の**VRRP優先度を低下**させます。その結果、優先度の高くなったL3SW32がVRRPマスタに昇格し、トラフィックがL3SW32経由に切り替わり、サービス継続性を確保できます。
**関連する重要な技術知識**:
-   **VRRP (Virtual Router Redundancy Protocol)**: 複数のルータを束ねて仮想ルータとして機能させるプロトコル。アクティブ/スタンバイ構成をとり、マスタールータに障害が発生した際にバックアップルータが引き継ぐことで、デフォルトゲートウェイの冗長化を実現します。
-   **障害トラッキング（トラック機能）**: ルータの物理インターフェースや特定のルートのダウンを検知し、VRRPの優先度を動的に変更する機能。

## 設問2

-   **設問5(4)**: 動作確認を行う目的について、単にリバースプロキシの動作確認だけでなく、移行手順全体の中で「正しく作業が行われたこと」を確認する視点を持つよう求めています。問題文中の状況をきちんと読み取り、**多角的な視点**で目的を把握する力が重要です。
