# 問1 解説

## 問題の概要
本問題は、D社の生活用品メーカーにおける在庫管理システムを題材に、データ分析のためのデータ抽出、データベース設計、運用処理について問うものです。具体的には、分析データ（Zチャート、ヒートマップ、在庫回転率）の提供依頼に対応するため、概念データモデルの理解、SQLを用いたデータ抽出・集計、計数格納処理の設計と検証、およびデータベース構造の変更が課題として与えられています。

## 出題のポイント
この問題では、データベーススペシャリストとして以下の知識・スキルが総合的に問われています。

1.  **概念データモデルの理解と業務ルールとの整合性判断**: 与えられた概念データモデル（ER図）が、示された業務ルールに合致しているかを判断する能力。特にエンティティタイプ間のリレーションシップ（カーディナリティ、オプション性）やスーパータイプ/サブタイプ関係の理解が重要です。
2.  **SQLによるデータ抽出・集計**: Zチャートやヒートマップの作成に必要なデータを、SQLの集計関数や**ウィンドウ関数**（`RANK() OVER()`など）を適切に用いて抽出・加工する能力。
3.  **データベース更新処理の設計**: 赤伝・黒伝処理を伴う在庫計数（商品有高表、残高集計表）の更新処理において、そのロジック（洗替え処理）や処理方式（同期/非同期）の設計思想、およびパフォーマンスへの影響を理解する能力。
4.  **データ検証と異常値分析**: データベースに格納されたデータの整合性や異常値を、業務知識に基づいて分析・判断する能力。
5.  **データベース構造変更（外部キー設計）**: 新たに導入するテーブル（受払明細、受払残高、残高集計）について、既存の概念データモデルとの整合性を保ちつつ、適切な**外部キー**を設定する能力。

採点講評では、SQLの集計関数やウィンドウ関数の理解不足、洗替え処理の設計思想、外部キーによる参照制約の理解が不十分な受験者が多かったことが指摘されており、これらのポイントを重点的に学習する必要があります。

- 棚307は「高」頻度、通路の左側、入口から比較的近いグループ。
- 棚604は「高」頻度、通路の右側、出口から比較的近いグループ。

なぜこの2つの棚なのでしょうか？
図の矢印は、入口から出口へ向かう主要な順路を示していますが、必ずしも全ての棚を通過するわけではありません。特に、入口側と出口側で並行する通路が複数あるため、順路が左右に分かれることがあります。

- 棚604は「高」頻度で、出口付近の右側の通路にあります。
この2つは「高」頻度でありながら、通路の左右に分かれ、入口から出口にかけて対角線上に位置しています。

考えられる意図は、作業順路において「高」頻度の棚が左右の通路に分散していることで、作業員が左右を行ったり来たりする**移動距離**が発生している点に着目しているのかもしれません。

- 307（高）は左通路の中盤。
- 604（高）は右通路の終盤。

これを解消するために、例えば307にある高頻度商品を600番台の「中」「低」頻度の棚と入れ替える、あるいは604にある高頻度商品を300番台の「中」「低」頻度の棚と入れ替える、といった工夫が考えられます。

しかし、設問は「二つの棚に配置されている商品を相互に入れ替えて」とあります。
307と604はどちらも「高」頻度なので、商品を入れ替えても、**高頻度であるという性質はそのまま**です。
考えられるのは、図の最短順路が示す「入口→左通路→奥→右通路→出口」または「入口→右通路→奥→左通路→出口」のようなパスを想定した場合、307（左中盤）と604（右終盤）が遠く離れていると、この二つの高頻度品を取る際に**長い移動距離が発生する**という問題です。
もし307と604を入れ替えることで、例えば604の商品を307の位置に、307の商品を604の位置に置くと、入口から遠い位置にある高頻度品が入口に近くなり、逆もまた然りです。これでは単純な効率化にはなりません。

もう一度、効率化の定義を考えると、「出庫作業における移動距離を短縮して効率化」です。
図4をよく見ると、300番台と600番台はそれぞれ「高」が点在しているものの、その間には「中」や「低」が多くあります。
特に、307（高）と604（高）は、それぞれが所属する通路の端の方に位置しているように見えます。

そこで、この二つの「高」頻度棚にある商品を入れ替えることで、**作業経路が集中している場所**に「高」頻度品を寄せる、あるいは**作業順路の早い段階**で「高」頻度品が取得できるようにする、という意図が考えられます。

-   307の位置には604の商品（高頻度）が置かれ、
-   604の位置には307の商品（高頻度）が置かれる。
これは直接的な移動距離の短縮にはならないように見えます。

しかし、もし**307と604がそれぞれ異なる作業経路の要所に位置している**と解釈するならばどうでしょうか。
例えば、入口から左側の通路を通る作業者と右側の通路を通る作業者がいて、それぞれ307と604が「高」であるため、両方の通路からアクセスされる頻度が高いとします。

仮に、307と604が異なる作業経路の端にあり、それらの商品を交換することで、より多くの作業経路で効率が向上すると考えられます。

-   **連携 WT 内の最大登録TS**: 処理WTに溜まった連携WTのデータのうち、最も新しい登録TSより古い、または同等の登録TSを持つデータのみを対象とする。

したがって、以下の条件が必要です。
1.  **登録 TS が処理 WT 内で最大の登録 TS 以下であること**: 処理WTに溜められた連携WTのデータは、計数格納処理が実行されるまでに連携WTに追加された全ての入出荷情報を含みます。したがって、計数格納処理の開始時点での最新の登録TS、つまり処理WT内の最大登録TSを上限とする。

これらの条件は、トランザクションの分離性とデータの一貫性を確保するために重要です。

# 問2 解説

## 問題の概要
本問題は、ドラッグストアチェーンF社の業務改革に伴う商品物流システムの概念データモデル及び関係スキーマの設計に関するものです。物流拠点、配送地域、店舗、商品カテゴリー、アイテム、商品、そして幹線ルート・支線ルートといった多岐にわたる業務要件を、ER図や関係スキーマとして適切にモデリングする能力が問われます。マスター/在庫領域とトランザクション領域に分けて設計する方針が示されています。

## 出題のポイント
この問題では、以下に示すデータベース設計の基礎から応用までが問われています。

1.  **業務要件の正確な理解と抽象化**: 大量の業務記述から、システム化に必要な**エンティティタイプ**、**属性**、およびそれらの間の**リレーションシップ**を正確に抽出し、抽象化する能力。
2.  **概念データモデルの設計**: 抽出した情報を基に、概念データモデル（ER図）を作成する能力。特に、スーパータイプ/サブタイプ、階層構造（商品カテゴリー）、多対多リレーションシップの解消といった高度なモデリング要素を適切に適用できるかが問われます。また、指定された表記ルールに従って図を完成させることも求められます。
3.  **関係スキーマの設計**: 概念データモデルを基に、**第3正規形**を満たす関係スキーマを設計する能力。主キー、外部キーを正確に特定し、表記ルール（実線下線、破線下線）に従って記述することが重要です。
4.  **マスターデータとトランザクションデータの分離**: データモデルをマスター/在庫領域とトランザクション領域に分けて設計する方針が与えられており、それぞれの領域に適切なエンティティを配置し、関係性を表現する能力。

採点講評では、マスター/在庫領域に関する問題の正答率は高かったものの、トランザクション領域に関する問題（設問1(3)および設問1(4)のチ〜ノ）の正答率が低かったことが指摘されています。これは、トランザクション領域のモデリングには、業務手順や情報の連鎖を深く理解する必要があるためです。業務フローをデータがどのように流れるかを具体的に想像しながらモデリングする訓練が求められます。

-   **外部キー (Foreign Key)**: あるテーブルの列が、別のテーブルの主キーを参照するものです。参照先の主キーと同じデータ型と制約を持ちます。破線の下線で示されます。ただし、外部キーが複合主キーの一部である場合は、実線の下線となります（問題文P4,P5の表記ルール参照）。

**誤答しやすいポイント**:
-   **正規化の不足**: 業務記述から抽出した属性を適切に正規化せず、冗長なデータ構造を作成してしまうと、第3正規形を満たしません。
-   **キーの指定ミス**: 主キーと外部キーの区別、複合主キーの構成、外部キーが主キーの一部である場合の表記ルール（破線下線がない）を正確に理解していないと誤答します。特にトランザクション領域では、各伝票が持つキーと、それが参照するマスターデータや他の伝票のキーの関係を正確に把握することが重要です。

## 採点講評からのアドバイス
採点講評では、マスター及び在庫の領域に関する問題は正答率が高かったものの、トランザクションの領域に関する問題は正答率が低かったことが指摘されています。これは、マスターデータが比較的静的な「資源」に関する情報であるのに対し、トランザクションデータは「業務手順」や「情報の連鎖」といった動的な側面を深く理解しないと正確にモデリングできないためです。

合格を目指すには、以下の点を意識して学習しましょう。
1.  **業務記述の徹底的な読み込み**: 単にキーワードを拾うだけでなく、業務がどのように行われ、どのような情報がいつ、どこで発生し、どのように引き継がれるのかを具体的にイメージする習慣をつけましょう。図や表だけでなく、テキスト記述の隅々まで注意深く読むことが重要です。
2.  **業務フローの可視化**: 複雑な業務フローは、図（フローチャートやシーケンス図など）を描いて可視化することで、情報の連鎖やエンティティ間の関係性をより明確に理解できます。
3.  **仮説と検証**: データモデル設計は一度で完璧にできるものではありません。仮説（このエンティティはこう関連するだろう）を立て、業務記述や設計方針と照らし合わせて検証する、というサイクルを繰り返すことが重要です。
4.  **正規化の徹底**: 第3正規形までの正規化ルールをしっかりと理解し、常に適用することを心がけましょう。これにより、データの整合性を保ち、更新異常を防ぐことができます。
5.  **主キー・外部キーの正確な設定**: どの属性がエンティティを一意に識別する主キーなのか、どの属性が他のエンティティを参照する外部キーなのかを正確に把握し、設計ルールに則って記述する練習を繰り返しましょう。特に複合主キーや、外部キーが主キーの一部を構成する場合の表記に注意が必要です。

## 関連する学習テーマ
本問題に関連して復習すべき学習テーマは以下の通りです。

-   **概念データモデリング**: ER図の作成、エンティティ、属性、リレーションシップ（カーディナリティ、オプション性）の理解。
-   **スーパータイプ/サブタイプ**: 階層構造を持つエンティティのモデリング手法。
-   **データベース正規化**: 第1正規形から第3正規形までのルール、関数従属、推移的関数従属の理解。
-   **関係スキーマ設計**: 概念データモデルから物理的なスキーマへの変換、主キー、外部キーの設計と表記。
-   **SQL (特にデータ操作言語 DML)**: SELECT文によるデータ抽出、GROUP BY句、集計関数、ウィンドウ関数（RANK() OVER()など）、CASE文による条件分岐。
-   **トランザクション管理**: トランザクションのACID特性、同期/非同期処理、排他制御、回復処理。
-   **データ分析の基礎**: Zチャートやヒートマップなど、各種グラフが示すデータの傾向の読み取り方。
-   **業務分析**: 業務要件定義、業務フロー分析、データフロー分析。

これらのテーマを深く学習し、実践的な演習を繰り返すことで、データベーススペシャリストとして求められる能力を向上させることができます。
